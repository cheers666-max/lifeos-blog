我日常同时开 4 个 AI coding agent：Claude Code、Codex、opencode、Gemini。它们都支持 "skill"——把一段提示词 + 工具定义 + 示例塞进一个文件夹，agent 就能调用。问题是，每个 agent 把 skill 放在**自己的目录**里，于是同一台电脑上，skill 被关进了四个互不相通的抽屉。

这篇记录我怎么把这团浆糊收拾干净。**一条主线：别让多个目录互相追平，让它们根本就是同一个目录。** 后面每一节都是这条主线上的一步。

## 先看清楚痛点：同一台电脑，skill 却互相看不见

四个 agent 的 skill 目录长这样：

```text
~/.claude/skills        # Claude Code 的
~/.codex/skills         # Codex 的
~/.config/opencode/skill
~/.gemini/skills
```

结果就是：我在 Codex 里写了个 `ralph` 工作流 skill，回到 Claude Code 死活找不到；在 CC 里调好的 `intuition-flywheel`，Codex 也看不见。同一台电脑、同一个我，skill 却被锁在四个抽屉里，钥匙还各不通用。

{{< figure src="01-four-drawers.png" alt="西高地白梗站在四个分开的抽屉中间，每个抽屉锁着自己的 skill，彼此用虚线隔开" caption="四个 agent 各存各的 skill，中间那只想从一个抽屉看进另一个，被虚线挡住" >}}

> **小结**：痛点不是"skill 不够多"，而是"同一份能力被切成了四份，互相不可见"。

## 我的第一反应是错的：互相镜像 + 定时同步

最先想到的方案，现在看很典型，我差点就这么干了：

1. 写个脚本，把每个文件夹里独有的 skill 软链到另外几个文件夹，让它们"长得一模一样"；
2. 包装成一个 `sync-skills` skill；
3. 再配个同名 alias；
4. 用 launchd 每天早上 10 点自动跑一次。

写到一半就发现不对劲：这是**互相镜像**。A 多了个 skill 要链到 B、C，B 多了又要链回 A、C……每个文件夹都变成"真文件 + 软链"的混合体，脚本还得分辨谁是本体。更要命的是——**正因为是镜像，才需要一个定时任务去追平**。而定时任务天然有延迟：我下午 2 点写的 skill，得等明天 10 点才在别的 agent 里生效。

一句话点醒我：

> **cron 是症状，不是解法。根因是"多份独立的目录"。**

## 正确的解法：一个真身，其余全是它

别让目录互相追平，让它们**根本就是同一个目录**。

选一个本体 `~/.skills`，把每个 agent 的 skills 目录整个软链过去：

```bash
ln -s ~/.skills ~/.claude/skills
ln -s ~/.skills ~/.codex/skills
ln -s ~/.skills ~/.config/opencode/skill
# ...
```

软链是同一个 inode。写一个 skill 丢进 `~/.skills`，4 个 agent **立刻**都看得到——没有脚本、没有 alias、没有 launchd、没有漂移。原计划的 4 步，直接砍到 0 步。

{{< figure src="02-mirror-vs-source.png" alt="左边西高地在多个盒子间复制、连出一团乱红线，还摆着闹钟；右边一个'一份真身'盒子，几根橙色箭头从小框指过来" caption="左：镜像要靠定时任务追平，永远在乱。右：一份真身 + 软链指针，零同步" >}}

这不是拍脑袋。后来去看了 cc-switch（一个管 Claude/Codex/Gemini 配置的工具）的 skill 管理页，它内部正是这个"单一真身 + 分发"的模型，只是它分发用的是**复制**而不是软链——所以它得再用 SHA-256 哈希去检测哪些 skill 变了、给你一个按钮追平。复制就要付同步的税；软链不用。复制的好处是能跨 Windows（Windows 软链 + 云盘会坏），但我只在 Mac/Linux，软链是更优解。

> **小结**：把"多份镜像"换成"一份真身 + 多个指针"，同步这件事从根上消失了。

## 落地时挖出的暗雷：先扫，再合

真正动手合并时，分析阶段挖出几个不动手根本不知道的坑：

- `~/.claude/skills` 里有 166 个真实 skill，但混着 **13 个死链**——它们指向一个早已不存在的 `.agents/skills`，其中一个甚至指向旧用户名 `/Users/yuajing/`。上一套配置的残骸，内容早没了。
- `~/.codex/skills` 有 31 个，其中 **30 个是 Codex 独有**的工具链 skill。
- 唯一一个重名冲突 `security-review`：Claude 版 495 行、还多带一个 `cloud-infrastructure-security.md`；Codex 版只有 300 行。**直接合并会覆盖丢版本。**

{{< figure src="04-merge-landmines.png" alt="西高地白梗举着探照灯在一堆 skill 卡片里翻找，一只爪子拎着断掉的线（死链），旁边两张同名卡片叠在一起打了红色感叹号" caption="合并前先拿探照灯扫一遍：拎出死链、标出重名冲突，先备份再动手" >}}

所以正确的顺序是：**先备份、再分析冲突、最后才替换**。我保留了 Claude 的超集版，Codex 版单独留底；死链全清掉；最后 166 + 30 = **196 个 skill** 进了 `~/.skills`。两个原目录换成软链，当场验证：在 Codex 目录下能看到 Claude 的 skill，反之亦然。

> **小结**：任何"合并目录"的操作，动手前先扫同名冲突和死链。`diff -rq` 和 `find -type l` 两条命令，省掉一场事故。

## 跨机器：内容走 git，软链各机器本地建

单机搞定了，多台机器怎么办？

有人会想到把整个文件夹丢进 Dropbox / OneDrive 同步。**别这么干**——这些云盘客户端跨平台时基本不保留软链，Windows 上还会直接搞坏。

正确做法是**内容走 git、软链各机器本地建**：

- `~/.skills` 本身设成一个 GitHub **私有**仓库；
- 改了 skill 就 `git push`，另一台 `git pull`；
- 软链不进 git，每台机器跑一次接入脚本就行。

{{< figure src="03-cross-machine-git.png" alt="西高地把写着 skill 的纸往 git 云里 push，另一台电脑从云里 pull 下来，旁边的西高地在本地用蓝线接软链；左下角一个被红叉划掉的网盘" caption="内容走 git（push / pull），软链各机器本地建。左下角那个网盘——别用它同步软链" >}}

接入脚本 `link-skills.sh` 做成**幂等**的：已经是正确软链就跳过，是真实目录就先备份再建链，agent 没装就跳过。新机器两步到位：

```bash
git clone <私库> ~/.skills
~/.skills/link-skills.sh
```

> **小结**：跨机器同步的是"内容"，不是"链"。git 管内容、本地建链，正好绕开云盘毁软链的坑。

## 一个安全插曲

把仓库推到 GitHub 这一步，我让 AI agent 代劳，结果被它自己的安全闸拦了三次——理由是"往一个你没亲口指名的外部仓库推整棵目录树"属于数据外发高危动作，哪怕是我自己的私库、哪怕加了 `git push` 允许规则也不放行。最后是**我自己**在终端敲 `git push` 才推上去的。

当时有点烦，回头想这个边界是对的：把一堆 skill（里面可能夹带内网 IP、token、密码）推到外部服务，本就该由人来按下最后那个按钮。所以也提醒一句：**私库归私库，提交前务必确认没有明文密钥**，敏感凭据走 Keychain / 环境变量 / `.env`（并在 `.gitignore` 里排除）。我推之前专门扫过一遍新增内容，确认没有硬编码密钥才推。

## 收尾：把主线再串一遍

> **同一份能力被四个 agent 切成四份、互相看不见 → 第一反应是让四份互相镜像，但那逼出一个永远在追平的定时任务 → cron 是症状，根因是"多份独立目录" → 把多份换成"一份真身 + 多个软链指针"，同步问题直接消失 → 合并前先扫死链和重名、先备份 → 跨机器让内容走 git、软链各机器本地建，别走云盘。**

| 层 | 做法 | 解决什么 |
|---|---|---|
| 单机扇出 | 各 agent 目录软链到 `~/.skills` | 4 个 agent 共享同一份 skill，零同步 |
| 跨机器 | `~/.skills` 设为 git 私库 | 内容版本化、可回滚 |
| 每台机器 | `link-skills.sh` 幂等建链 | 软链不走 git，本地建一次永久有效 |

最值钱的不是哪条命令，是那句话：**当你发现自己在写定时同步任务，先停下来问一句——是不是因为我维护了多份本该是一份的东西？** 把"多份镜像"换成"一份真身 + 多个指针"，同步问题往往直接消失。
